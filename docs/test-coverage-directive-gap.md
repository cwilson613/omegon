# Test coverage directive gap — agents consistently omit tests unless explicitly forced

## Overview

Agents running within Omegon consistently produce code without test coverage. Tests are written only when the operator explicitly requests them (adversarial assessment → "O1: no test coverage" → operator says "fix all omissions" → tests get written). This is a systemic directive gap, not an individual agent failure.

Observed pattern in this session alone:
- Mind-per-directive lifecycle: 4 files changed, 0 tests written. Tests added only after adversarial review.
- Lifecycle gate ergonomics: 3 files changed, existing tests updated to match new behavior but no NEW tests for new functions (extractAndArchiveDesignSpec). Added only after review.
- Cleave children (4 parallel): only 1 of 4 wrote tests (mind-chain-tests, which was explicitly scoped as a test-writing task).
- Dashboard directive indicator: 0 tests. Added only after review.
- Branch↔mind consistency: 0 tests. Added only after review.

## Research

### Full directive chain audit — where tests are mentioned

Traced every source of agent instructions from system prompt through AGENTS.md through skills through cleave child contracts.

**Layer 1: Pi system prompt** (`vendor/pi-mono/.../system-prompt.ts`)
- Zero mentions of testing. The built-in system prompt covers tools, guidelines (be concise, show file paths), and pi documentation links. No testing guidance whatsoever.

**Layer 2: Omegon AGENTS.md** (`config/AGENTS.md`, deployed to `~/.pi/agent/AGENTS.md`)
- One mention: "writing tests before production code" — buried in a parenthetical inside step 4 of the OpenSpec lifecycle description
- The Development Methodology section says "spec → test → code → verify" in the overview but never expands on what "test" means or when it's mandatory
- No standalone testing directive, no test-first requirement, no "every function must have tests" rule
- The Completion Standards section says "work is not done until committed and pushed" but says nothing about test coverage

**Layer 3: TypeScript skill** (`skills/typescript/SKILL.md`)
- Has a "Testing with node:test" section with conventions
- But skills are LAZY-LOADED — the agent must `read` the skill file, and the system prompt says "Use the read tool to load a skill's file when the task matches its description"
- The skill description says "testing with node:test" but the agent has no directive to proactively load it
- Even when loaded, the conventions say "test behavior, not implementation" but don't say "write tests for every change"

**Layer 4: Cleave child contract** (`extensions/cleave/dispatcher.ts:289-298`)
- Rule 5: "Verification: Run tests or checks and report results in the Verification section."
- This says RUN tests, not WRITE tests
- The task file template has a "Verification" section but it's a blank field to fill in, not a mandate

**Layer 5: OpenSpec tasks.md** (generated by `/opsx:ff`)
- Includes task items like "Write tests for X" for each requirement
- But these are checklist items that cleave treats as optional subtasks within a group
- A child can mark a task group as SUCCESS without completing every checkbox

**Layer 6: Cleave review** (when `review: true` is passed to `cleave_run`)
- The adversarial reviewer checks for bugs and spec compliance
- Does NOT check for test coverage — it reviews what exists, not what's missing

**The gap**: No layer in the directive chain says "you must write tests for new code." The closest is a parenthetical in AGENTS.md and a "run tests" instruction in the cleave contract. The agent's default behavior (from training) is to write code that works, not code that's tested. Without an explicit directive, tests are treated as optional.

### Where test directives should be injected

The fix needs to operate at multiple layers — a single mention isn't enough because agents process context hierarchically and tend to weight explicit, prominent directives over buried ones.

**Fix 1: AGENTS.md — add a standalone testing directive (highest authority)**

Add a top-level section to `config/AGENTS.md` alongside Attribution Policy, Development Methodology, and Completion Standards:

```markdown

## Decisions

### Decision: Multi-layer test directive injection: AGENTS.md + cleave contract + review check

**Status:** exploring
**Rationale:** A single mention isn't enough — agents weight prominent, repeated directives. The fix needs: (1) AGENTS.md top-level Testing Standards section (highest authority, all sessions), (2) cleave child contract rule change from "run tests" to "write and run tests", (3) cleave review test-coverage check as a Warning-level finding. These three layers reinforce each other without requiring the agent to proactively load a skill file.

## Open Questions

*No open questions.*

## Testing Standards

**Every non-trivial code change must include tests.**

- New functions, methods, or modules require test coverage
- Modified behavior requires updated or new tests that verify the change
- Bug fixes require a regression test that would have caught the bug
- Test files are co-located: `foo.ts` → `foo.test.ts`
- Tests run before commit: `npm test` (or project equivalent) must pass
- Untested code is incomplete code — do not commit without tests

### When tests may be skipped
- Config files, documentation, type-only changes
- One-line fixes where existing tests already cover the behavior
- Design documents and OpenSpec artifacts
```

**Fix 2: Cleave child contract — add WRITE tests rule**

Change rule 5 in `dispatcher.ts` from:
```
5. **Verification**: Run tests or checks and report results in the Verification section.
```
To:
```
5. **Testing**: Write tests for new functions and changed behavior. Run them and report results in the Verification section. Untested code is incomplete.
```

**Fix 3: Cleave review — check for test coverage**

When `review: true`, the adversarial reviewer should include a test coverage check:
- For each new/modified `.ts` file, check if a corresponding `.test.ts` exists
- Flag new exported functions without test coverage as a Warning

**Fix 4: OpenSpec tasks.md — make test tasks mandatory, not optional**

Change the task generation template so test items are NOT sub-items of implementation tasks but separate required tasks:
```

## 1. Implement feature X

- [ ] 1.1 Core implementation
- [ ] 1.2 Edge case handling

## 2. Test feature X (REQUIRED)

- [ ] 2.1 Unit tests for core behavior
- [ ] 2.2 Edge case and error path tests
```

This makes test tasks their own cleave children rather than checkboxes within implementation children.

**Fix 5: System prompt promptGuidelines — add testing guideline**

Add to the tool `promptGuidelines` array in the cleave extension or as a global guideline:
```
"Write tests for new code — co-locate as *.test.ts, run with npm test"
```

This ensures the guideline appears in every session's system prompt, not just when a skill is loaded.
